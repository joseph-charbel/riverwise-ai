# Riverwise Server

The Riverwise back-end is a FastAPI application that provides the API consumed by the front-end client. Its primary responsibility — once fully integrated — is to accept information card text from a hotspot interaction and return an AI-enriched, pedagogically adapted explanation generated by a Groq-hosted LLM. The AI layer uses [LangChain](https://python.langchain.com/) with a configurable system prompt that adapts the response to a student's grade level and personal interests.

Currently the production-bound endpoint (`POST /api/dummy-invoke`) returns a stub response. The real `Model` class is implemented and tested; connecting it is a one-file change described in [Swapping the stub for the real LLM](#swapping-the-stub-for-the-real-llm).

For the full-stack setup, see the [root README](../README.md).

## Installation and development

```bash
cd server

# 1. Copy the environment template and fill in your Groq API key
cp .env.example .env

# 2. Install dependencies (reads uv.lock for a reproducible install)
uv sync

# 3. Start with hot-reload
uv run uvicorn app.api:app --reload --host 0.0.0.0 --port 8000
```

Alternatively, use `main.py` which exposes both launch modes:

```bash
uv run python main.py --api    # starts uvicorn on :8000 (no hot-reload)
uv run python main.py          # runs the CLI test harness (app/index.py)
```

The server will be available at `http://localhost:8000`. Swagger UI is available at `http://localhost:8000/docs`.

## Environment variables

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `GROQ_API_KEY` | Yes (for real LLM) | — | API key from [console.groq.com](https://console.groq.com). Without it, only the stub endpoint works. |
| `CORS_ORIGINS` | No | `http://localhost:3000,http://localhost:5173` | Comma-separated list of origins the browser is allowed to call from. Add your deployment domain here. |
| `LOG_LEVEL` | No | `INFO` | Python logging verbosity: `DEBUG`, `INFO`, `WARNING`, `ERROR`. |

All variables are loaded from `server/.env` at startup via `python-dotenv`. The `.env` file is git-ignored and must be created manually from `.env.example` — never commit secrets.

## API reference

### `GET /health`

Liveness check. Returns immediately with no side effects.

**Response `200 OK`:**
```json
{ "status": "ok" }
```

---

### `POST /api/dummy-invoke`

Accepts a text prompt and returns a response. Currently returns a stub that echoes the prompt with `DUMMY LOGIC` markers. This endpoint is called by `InfoHotspot` in the client when a player clicks an info hotspot.

**Request body:**
```json
{ "prompt": "<information card body text>" }
```

**Response `200 OK`:**
```json
{ "content": "<AI or stub response string>" }
```

**Current stub behaviour:** wraps the prompt as:
```
DUMMY LOGIC
<prompt>
DUMMY LOGIC
```

**Intended real behaviour:** passes the prompt to `Model.invoke()` alongside `grade_level` and `student_interest`, and returns the LLM's pedagogically adapted explanation.

## Architecture

```
server/
├── main.py                      Entry point — selects API or CLI mode
├── app/
│   ├── api.py                   FastAPI app, CORS, route definitions
│   ├── index.py                 CLI test harness + dummy_invoke stub
│   ├── logging_config.py        Centralised logging setup
│   └── ai/
│       └── model.py             Model class — wraps ChatGroq via LangChain
│   └── config/
│       ├── base_config.py       Pydantic config models, ConfigManager singleton
│       └── config_loader.py     YAML loader, placeholder interpolation
└── config/
    └── config.yaml              LLM model selection, sampling params, system prompt
```

### Config loading sequence

The configuration system initialises itself at import time. Understanding this order matters if you add new modules:

```
config_loader.py imported
  → load_dotenv() runs          (.env is read into os.environ)
  → config/config.yaml is read  (YAML → Python dict, stored as CONFIG)
  → render_placeholders()       (optional env-var interpolation in YAML strings)

base_config.py: ConfigManager (singleton) instantiated on first access
  → reads CONFIG dict
  → reads GROQ_API_KEY from env   (stored as SecretStr)
  → builds ChatConfig
  → builds PromptConfig
```

### `app/api.py`

The FastAPI application. Reads `CORS_ORIGINS` from the environment and configures `CORSMiddleware`. All credentials, methods, and headers are allowed. Defines the `DummyInvokeRequest` and `DummyInvokeResponse` Pydantic models for the `/api/dummy-invoke` route.

### `app/ai/model.py`

The `Model` class wraps `ChatGroq` from LangChain. On construction it:
1. Reads `ChatConfig` from `ConfigManager` (model name, API key, extra kwargs such as `temperature`).
2. Reads `PromptConfig` to get the system prompt template.
3. Instantiates `ChatGroq` with the configured model and kwargs.

Its `invoke(prompt, *, grade_level, student_interest)` method renders the system prompt template (replacing `{{grade_level}}` and `{{student_interest}}`), then sends a two-message list to the LLM: a `SystemMessage` and a `HumanMessage` containing the prompt.

### `app/config/base_config.py`

Defines two Pydantic models:

**`ChatConfig`**

| Field | Type | Source |
|-------|------|--------|
| `provider` | `str \| None` | `config.yaml → chat_model_config.provider` |
| `api_key` | `SecretStr` | `GROQ_API_KEY` env var |
| `model` | `str` | `config.yaml → chat_model_config.model` |
| `kwargs` | `dict` | `config.yaml → chat_model_config.kwargs` |

**`PromptConfig`**

| Field | Type | Source |
|-------|------|--------|
| `system_prompt` | `str` | `config.yaml → prompts_config.system_prompt` |

`ConfigManager` is a singleton (enforced via `__new__`) so the YAML file is only parsed once per process.

### `app/config/config_loader.py`

Loads `config/config.yaml` relative to the project root and exposes the result as the module-level `CONFIG` dict. Supports optional placeholder interpolation: any string value in the YAML that contains `{env_var_name}` will be replaced with the corresponding environment variable value at load time.

### `app/logging_config.py`

Provides `get_logger(name)`, which every module calls instead of `logging.getLogger` directly. On first call it invokes `setup_logging()`, which configures the root logger with level from `LOG_LEVEL` and the format:

```
%(asctime)s | %(levelname)s | %(name)s | %(message)s
```

## AI model configuration

All tunable LLM parameters live in `config/config.yaml`. No code changes are needed to adjust the model or the system prompt.

```yaml
chat_model_config:
  provider: groq
  model: llama-3.1-8b-instant   # ← change model here
  kwargs:
    temperature: 0              # ← add/change sampling params here

prompts_config:
  system_prompt: |
    ### Role
    You are an elite private tutor specializing in adaptive pedagogy...
    ...
    Grade level: {{grade_level}}
    Student interest: {{student_interest}}
```

### Change the LLM model

Replace `chat_model_config.model` with any model ID available on [Groq's model list](https://console.groq.com/docs/models), for example:

```yaml
model: llama-3.3-70b-versatile
```

Any extra keys under `kwargs` are passed directly to the `ChatGroq` constructor, so sampling parameters like `temperature`, `max_tokens`, or `top_p` can be added here without code changes.

### Update the system prompt

Edit `prompts_config.system_prompt`. The prompt is a multi-line YAML string. Two dynamic placeholders are substituted at runtime:

| Placeholder | Value | Source |
|-------------|-------|--------|
| `{{grade_level}}` | Student's school grade (e.g. `"3"`) | passed to `Model.invoke()` |
| `{{student_interest}}` | Student's interest topic (e.g. `"Space"`) | passed to `Model.invoke()` |

The current prompt instructs the model to act as an adaptive private tutor for children, use interest-based analogies, apply a Socratic style, and structure every response as: **The Hook → The Lesson → The Check-In**.

## Swapping the stub for the real LLM

The real `Model` class is implemented and unit-testable via the CLI (`uv run python main.py`). Connecting it to the API requires two small changes in `app/api.py`:

**1. Import `Model` instead of (or alongside) `dummy_invoke`:**

```python
# app/api.py — replace
from app.index import dummy_invoke

# with
from app.ai.model import Model
model = Model()
```

**2. Update the route handler to call `model.invoke()`:**

```python
@app.post("/api/invoke", response_model=DummyInvokeResponse)
async def invoke(request: DummyInvokeRequest):
    result = model.invoke(
        request.prompt,
        grade_level=request.grade_level,       # add these fields to the request model
        student_interest=request.student_interest,
    )
    return DummyInvokeResponse(content=result.content)
```

You will also want to extend `DummyInvokeRequest` with `grade_level: str` and `student_interest: str` fields, and update the client's `InfoHotspot` to send them (currently it posts only `{ prompt: body }`).

## Production build and Docker

The `Dockerfile` produces a minimal image containing only the application code and its runtime dependencies — no virtual environment, no dev tools, no secrets.

```bash
docker build -t riverwise-server .
docker run -p 8000:8000 \
  -e GROQ_API_KEY=your_key_here \
  -e CORS_ORIGINS=https://your-domain.com \
  riverwise-server
```

**Build stages:**

1. Starts from `python:3.12-slim`.
2. Installs `uv` via `pip`.
3. Copies `pyproject.toml` and `uv.lock`; runs `uv sync --frozen --no-dev` — installs only production dependencies, locked to exact versions.
4. Copies `app/` and `config/` directories.
5. Exposes port `8000`.

`main.py` and `.env` are intentionally excluded from the image (see `.dockerignore`). Secrets must be injected at runtime via environment variables, either through `docker run -e`, Docker Compose `environment:` / `env_file:`, or a secrets manager.

The start command is:

```
uv run uvicorn app.api:app --host 0.0.0.0 --port 8000
```
